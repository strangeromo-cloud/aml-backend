# AML 系统约束条件清单：检查逻辑与数据需求

> 按检查发生的层次整理：付款级硬规则 → 供应商级行为信号 → 加权评分 → 流程/治理约束。
> 每条给出：**判定逻辑**（demo 当前实现 + 生产化时的真实逻辑）和**数据支持**（要什么数据、来自哪个系统）。

---

## 一、入口 A：付款级高精度规则（命中任意 1 条 → T1，24h SLA）

这 4 条是 Legal 经验中"准确率 100%"的确定性规则，命中后约 30% 是真洗钱（PPV）。它们**跳出加权平均**，不被其他维度稀释。

### A1. 银行与营业地国家不一致（bankCountryMismatch）

| | |
|---|---|
| **判定逻辑** | 供应商收款银行账户所在国 ≠ 供应商注册/营业所在国 |
| **demo 代理实现** | `riskTypes 含 geographic 且 total ≥ 75` |
| **数据支持** | ① 收款账户国别 —— 付款系统（LGAP/S4）里的银行主数据，或从 IBAN/SWIFT BIC 前缀解析；② 供应商注册国、营业地 —— 供应商主数据（SAP/Ariba 供应商档案）；③ 中国供应商可用天眼查/企查查工商注册地交叉验证 |

### A2. 付款方与发票来源国不一致（payerInvoiceMismatch）

| | |
|---|---|
| **判定逻辑** | 发票开具方所在国 ≠ 合同/付款对手方所在国（钱要去的地方和发票来的地方对不上） |
| **demo 代理实现** | `riskTypes 含 geographic 且 reasonable = Unreasonable` |
| **数据支持** | ① 发票开票方国别 —— AP 应付系统 / 发票 OCR 抽取字段；② 付款主体（company code）所在国 —— S4；③ 合同签约方国别 —— 合同管理系统 |

### A3. 主营业务与采购品类不符（businessScopeMismatch）

| | |
|---|---|
| **判定逻辑** | 供应商工商登记的经营范围覆盖不了本次 PO 的采购品类（如注册"日用品贸易"的公司开"专业咨询服务"发票） |
| **demo 代理实现** | `riskTypes 含 docMissing` |
| **数据支持** | ① 经营范围 —— 天眼查/企查查（中国）或当地工商登记（海外）；② PO 采购品类 —— Ariba；③ 一张"经营范围 ↔ 品类"的映射表（需要自建，LLM 可辅助做语义匹配） |

### A4. 虚假发票特征（fakeInvoice）

| | |
|---|---|
| **判定逻辑** | 发票本身存疑：查验不通过、发票号重复、模板/字段异常、连号开票等 |
| **demo 代理实现** | `riskTypes 含 behavior 且 reasonableScore ≥ 60` |
| **数据支持** | ① 发票查验结果 —— 税务局增值税发票查验平台（中国）；② 发票号去重 —— AP 系统历史发票库；③ 发票影像与元数据 —— 发票 OCR/影像系统 |

## 二、入口 B：供应商级行为信号（无硬命中但供应商有信号 → T2，3 天 SLA）

单笔看正常、拉到供应商维度才暴露的模式。**必须有该供应商的全量付款流水**才能算。

### B1. 拆分付款（structuring，化整为零）

| | |
|---|---|
| **demo 逻辑** | 该供应商付款 ≥ 3 笔，且其中小额（< $100,000）≥ 3 笔 |
| **生产逻辑** | 滚动窗口（如 30 天）内 N 笔低于审批阈值的付款，合计超过阈值 —— 拆单规避审批的特征 |
| **数据支持** | 供应商全量付款流水（金额、日期、审批链）—— LGAP/Ariba/S4；内部审批阈值配置 —— 财务制度 |

### B2. 付款突增（surge，新供应商异常放量）

| | |
|---|---|
| **demo 逻辑** | 该供应商付款笔数 ≥ 4 |
| **生产逻辑** | 新供应商（入库时间短）付款频率/金额相对基线突增 |
| **数据支持** | ① 供应商入库/首单日期 —— 供应商主数据；② 付款时间线 —— 付款系统；③ 同品类供应商基线分布 —— 本系统自算 |

### B3. 高精度规则多次命中（cumulativeAlerts，汇总信号）

| | |
|---|---|
| **demo 逻辑** | 该供应商名下 ≥ 2 笔付款命中过入口 A 规则 |
| **性质** | 这是入口 A 的 roll-up（标"汇总"），不是新信号 —— 单笔已复核放行，但同一供应商反复触发本身就是信号 |
| **数据支持** | 本系统自身的告警历史（signal_rows / 告警记录表） |

## 三、T3 加权评分（无任何命中的付款，按分排序）

**总分 = 供应商风险分 × 40% + ML 付款分 × 35% + 合理性分 × 25%**（只用于 T3 排序；T1/T2 不看分）

### C1. 供应商风险分（vendorScore，0–100）

由以下因子加权（demo 中每个供应商带因子命中明细）：

| 因子 | 检查内容 | 数据支持 |
|---|---|---|
| Watchlist/制裁名单 | 实体/受益人命中制裁或观察名单 | OFAC/UN/EU 名单、Dow Jones、OpenSanctions |
| 离岸名单 | 注册于 BVI/巴拿马/开曼等离岸辖区 | 供应商注册国 + 离岸辖区清单 |
| 负面媒体情绪 | 腐败/欺诈/制裁相关负面报道 | Dow Jones Adverse Media / 媒体监控 |
| Basel AML Index | 注册国反洗钱风险指数 | Basel Institute 年度指数 |
| FATF | 注册国在 FATF 灰/黑名单 | FATF 官方名单 |
| 腐败感知指数 CPI | 注册国腐败风险 | Transparency International |
| 国有企业关联 | 股权含国资背景（PEP 风险放大器） | 天眼查/企查查股权穿透 |

### C2. ML 付款分（paymentScore）

单笔付款特征的模型分：金额分位、币种、路径、时间特征等。
**数据支持**：付款交易明细（LGAP/S4）+ 历史付款分布（自建特征库）。

### C3. 合理性分（reasonableScore：Reasonable / Warning / Unreasonable）

金额是否在合同/服务范围内、该供应商在此品类是否有已批准的采购历史。
**数据支持**：合同金额区间（合同系统）、PO 历史（Ariba）、品类基准价。

## 四、六个风险类型维度（riskTypes —— 因子标签，供上述规则和 chatbot 分析引用）

| 维度 | 覆盖内容 | 数据支持 |
|---|---|---|
| geographic 地理 | 高风险辖区、离岸、国别不一致 | 国别名单 + A1/A2 的国别数据 |
| pep 政治人物 | 股东/受益人为 PEP | Dow Jones PEP 库 + 股权穿透（天眼查 UBO） |
| adverseMedia 负面媒体 | 负面报道 | Dow Jones / 媒体监控 |
| docMissing 单据缺失 | 无 PO、无合同、业务不符 | Ariba/合同系统的单据完整性 |
| timeAnomaly 时间异常 | 周末/节假日付款、新供应商突增 | 付款时间戳 + **节假日日历** |
| behavior 行为异常 | 账户频繁变更、可疑路径、假发票 | 银行账户变更历史 + 发票查验 |

## 五、流程与治理约束（不是检测规则，但系统必须强制执行）

| 约束 | 逻辑 | 数据支持 |
|---|---|---|
| **SLA 超时** | T1 未复核 > 24h、T2 > 3 天 → 标超时 | 告警创建时间 + 复核时间戳 |
| **队列去重** | 待复核队列排除已复核付款；T2 付款由供应商条目代表，不重复列 | reviewedCases（复核记录，Postgres `review_records`） |
| **月度校准** | 按序：① 样本 < 4 → 待验证 ② 样本 ≥ 4 且 Wilson 95% 下界 ≥ 8% → 已验证 ③ 样本 ≥ 4 且下界 < 8% → 降级建议 | 每信号的触发数/复核数/确认数（signal_rows）+ 复核结论 |
| **反馈闭环** | 复核提交 → 状态"待模型反馈"；跑月度校准 → 翻为"已反馈" | review_records |
| **进化闸门（合规底线）** | Agent 发现的新信号只能进"候选"，必须过校准 + 人工审批才影响 Tier；**任何自动化不得直接改 Tier** | 候选信号表 + 审批记录（审计留痕） |
| **调查 playbook 强化** | 复核"确认欺诈"→ 命中的调查套路 confirmed +1；未覆盖特征 → 长出新套路 | playbook_entries + 复核结论 |

候选信号示例（当前 demo 里 Agent 发现的）：**大额 + 地理/行为组合** —— `金额 ≥ $200,000 且 (geographic 或 behavior)`，付款级，待校准验证。

## 六、数据需求总表（按来源系统）

| 来源系统 | 需要的数据 | 支持哪些检查 |
|---|---|---|
| **LGAP / S4（付款）** | 付款流水：金额、日期、币种、付款主体、收款账号 | A1、B1、B2、C2、时间异常 |
| **Ariba（采购）** | PO 品类、合同关联、供应商 onboarding、审批链 | A3、B2、C3、docMissing |
| **银行主数据 / SWIFT** | 收款账户国别（IBAN/BIC）、账户变更历史 | A1、behavior |
| **AP / 发票系统 + 税务查验平台** | 发票开票方、发票号、查验结果、影像 | A2、A4 |
| **供应商主数据** | 注册国、营业地、经营范围、入库日期 | A1、A3、B2 |
| **天眼查 / 企查查** | 工商信息、经营范围、股权穿透、受益人 UBO | A3、C1（国企）、PEP、调查取证 |
| **Dow Jones / OpenSanctions / OFAC** | 制裁、watchlist、PEP、负面媒体 | C1、PEP、adverseMedia、调查取证 |
| **Basel / FATF / TI CPI** | 国家层面风险指数与名单 | C1 国别因子 |
| **节假日日历** | 各国节假日 | timeAnomaly |
| **HR 系统**（待接） | 员工信息 | 员工-供应商关联（利益冲突，待做） |
| **本系统 Postgres** | 复核记录、信号台账、playbook | B3、校准、PPV、闭环全部 |

## 七、demo 与生产的差距提示

demo 里入口 A 的 4 条规则是用合成数据的 `riskTypes` 标签**代理**的（如 A1 用 "geographic + 总分≥75" 近似）。生产化时要换成上表的真实字段比对 —— 逻辑框架（双入口 + Tier + 校准闸门）不变，变的只是每条规则的取数来源。
