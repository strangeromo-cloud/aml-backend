# AML 付款风险识别系统 — 说明文档

> 面向 Legal / 合规团队的付款欺诈与洗钱风险识别系统原型。
> 本文档梳理系统的设计逻辑、功能模块、技术架构、创新点，以及后续需要对接的工作。

---

## 一、项目概述

本系统帮助 Legal（法务）部门从海量进行中的付款里，**高效、可解释地识别出有洗钱/欺诈风险的付款**，并形成"识别 → 调查 → 裁决 → 学习"的闭环。

| 项 | 内容 |
|---|---|
| 形态 | 单文件 HTML 演示前端 + 独立 LLM 后端服务 |
| 数据 | 25 笔合成付款、8 个供应商（演示数据集）|
| 语言 | 中英文双语 |
| 在线后端 | `https://aml-p.zeabur.app`（Zeabur · FastAPI · GPT-5.5）|

---

## 二、业务背景与核心挑战

真实业务有三个硬约束，决定了整套设计：

| 约束 | 数字 | 后果 |
|---|---|---|
| 付款量 | 约 10,000 笔/天（≈365 万/年）| 海量 |
| 实际洗钱案 | < 10 起/年 | 阳性率约 1/400,000，极端稀少 |
| ICAC（廉政公署）| 不共享其检测逻辑 | 没有权威标签 |
| 评分因素/权重 | 来自 Legal 经验 | 规则是唯一可靠资产 |
| 已有优势 | 准确率 100% 的规则，识别后约 **30% 是真洗钱（PPV）** | 行业稀缺（通常仅 1-5%）|

**由此推导出三条铁律：**

1. **传统监督式机器学习不可行** —— 一年凑不出 10 个正样本，没法训练模型。
2. **不能用准确率/召回率衡量** —— 不知道真实洗钱总数，无法量化漏报。
3. **目标改为 Precision@K** —— 让 Legal 每天能审的那 K 笔，含金量最高。

---

## 三、核心设计理念

> 不追求"识别所有洗钱"（做不到、无法验证），而追求"**在有限审核产能里，被审到的样本含金量最高**"。

关键决策：**分层告警（Tier）而非单一加权分**。因为加权平均会稀释强信号 —— 一笔命中两条 30% PPV 规则的付款，被其他正常维度一平均就沉底了。所以高精度信号必须"跳出"平均池。

## 四、检测模型：双入口 + 三层分级

### 4.1 六大风险维度（对应业务规则表）

`geographic 地理风险` · `pep PEP 人物` · `adverseMedia 不利媒体` · `docMissing 文件缺失` · `timeAnomaly 时间异常` · `behavior 行为指标`

### 4.2 双入口

```
入口 A —— 付款级（单笔触发）              入口 B —— 供应商级（聚合才显现）
4 条高精度规则（每条 ~30% PPV）：         3 个行为信号：
 • 银行/营业地国家不一致                   • 拆分付款（化整为零）— 原生
 • 付款方/发票来源国不一致                 • 付款突增（新供应商）— 原生
 • 主营业务 ≠ 采购品类                     • 高精度规则多次命中 — 汇总(入口A的roll-up)
 • 虚假发票特征
```

> **入口 A** 抓单笔异常；**入口 B** 抓跨笔模式（拆分、突增），是入口 A 看不到的。
> "高精度多次命中"是入口 A 向 vendor 层的**汇总**（标注"汇总"），把"某笔有问题"升级为"这家供应商系统性可疑"，不是新检测。

### 4.3 三层告警（Tier）

| Tier | 触发条件 | 推断 PPV | 复核时限（SLA）|
|---|---|---|---|
| **T1 紧急** | ≥2 条高精度规则；或 拆分+旗下高精度命中 | 50-70% | 24 小时内 |
| **T2 告警** | 1 条高精度规则；或其他供应商信号 | ~30% | 3 天内 |
| **T3 评分** | 无高精度命中 | 1-5% | 批量，加权分排序取 top K |

### 4.4 双入口汇入一个队列

付款级和供应商级告警**混排在同一份工作清单**，按 Tier 排序，每条标注来源（付款/供应商）。
**典型例子**：Aster Global 的单笔付款都是 T2/T3（金额小、不起眼），但 vendor 层因"拆分 + 多次高精度命中"升到 **T1** 置顶 —— 这正是 payment 维度漏掉、vendor 维度抓到的洗钱。

## 五、评分体系：主从关系

```
Tier 是主分类（决定先看谁、多急）
加权 score 是从属（只在 T3 灰色地带当排序依据）
```

| 评分元素 | 角色 |
|---|---|
| 加权总分 = 供应商分×40% + ML付款分×35% + 合理性分×25% | **仅 T3 排序** |
| ML 付款分 | T3 盲区覆盖主力（抓规则没覆盖的） |
| 6 风险维度 | 派生高精度规则 + 解释性 |

**界面体现**：付款列表中 T1/T2 行的分数列显示"—"；案例详情 T1/T2 只展示 Tier + 触发规则，T3 才显示加权分拆解。

## 六、完整闭环

```
┌──────────────────────────────────────────────────────┐
│ ① 检测+定级 → ② 调查 → ③ 裁决 → ④ 学习 ──┐          │
│      ↑                                      │          │
│      └────── 实测 PPV / 新信号 回流校准 ─────┘          │
└──────────────────────────────────────────────────────┘
```

| 段 | 做什么 | 对应模块 |
|---|---|---|
| ① 检测+定级 | 双入口 → Tier → 统一队列 | 法务工作台（待复核）/ 付款列表 |
| ② 调查 | 多源取证、AI 综合调查包 | 调查 Agent（调查并复核） |
| ③ 裁决 | Legal 复核、定结论 | 案例详情 / 法务工作台 |
| ④ 学习 | 测每个信号 PPV、升降级、发现新信号 | 信号台账 + 校准 |

**环的灵魂**：④ 测出的 PPV 正是 ① 分 Tier 的依据。没有 ④ 是静态拍脑袋；有了 ④ 随反馈自动校准。

## 八、信号台账与持续优化（系统的"进化引擎"）

### 8.1 实时 PPV
每次复核都结构化记录 `{付款, 结论}`。信号台账实时算：
**PPV = 确认洗钱数 ÷ 已复核数**。

### 8.2 每月校准（治理动作）
点"跑本月校准"→ 生成变更建议报告（含**置信区间可视化** + **判定规则说明**），人工审批后生效。

**判定按顺序执行（先样本关、再 PPV 关）：**

| 顺序 | 条件 | 结果 |
|---|---|---|
| ① | 已复核 < 4 笔 | **待验证**（样本太少，不下结论，继续观察。即便 PPV 0% 也可能是运气差）|
| ② | 已复核 ≥ 4 且 Wilson 95% 置信下界 ≥ 8% | **已验证**（信得过，保留为正式触发器）|
| ③ | 已复核 ≥ 4 且 Wilson 95% 置信下界 < 8% | **已降级**（数据足够且确实不准 → 评分因子）|

**核心原则**：「降级」是负面判决，需足够证据；样本不足时宁可暂缓观察，也不因小样本运气冤枉信号。

**界面**：校准报告每行用彩色徽章展示**状态 + 对应 Tier 的前后变化**，例如：
- 主营业务不符：`[已验证·T2] → [已降级·评分因子]`
- 拆分付款（3 笔）：`[待验证·观察] · 无变更`
- Agent 发现的信号（校准后）：`[待验证·观察] → [已验证·T2]`

状态→Tier 映射：已验证按 PPV 映射 T1(≥50%)/T2(≥15%)/T3；已降级=评分因子；待验证=观察（不触发）。

**为什么用置信下界**：防止被"样本少、碰巧好看"的点值忽悠。点值 60% 但样本 5 笔 → 下界 23%（达标）；点值 25% 样本 4 笔 → 下界 5%（不达标，降级）。两个表面相近的命中率，结局相反，区别全在样本数。

> 8% 阈值的含义：相对随机基线（约 0.00025%）提升 3 万倍以上，才值得占用 Legal 产能。（demo 设定，生产可调）
> 显示门槛与校准门槛已统一为 4 笔，避免"显示已验证但校准退回待验证"的假变更。

## 九、调查 Agent（Hermes 风格）

点案例详情"调查并复核"→ 弹出合并流程：

```
① gather_evidence()【模拟】
   按 vendor 国家路由：CN→天眼查/企查查；ROW→Dow Jones
   按付款 riskTypes/信号生成多源证据
② GPT-5.5 综合【真实】
   OpenAI Responses API 流式，reasoning + 调查结论
③ 给出建议：escalate / investigate / clear
④ 复核表单：结论按建议预选、原因预填证据
⑤ 一次提交 → 落库 + 更新 PPV
```

- **确认欺诈 + escalate** → 付款打 `已冻结` 标签（模拟暂停付款）
- 真实性：**LLM 综合是真的 GPT 调用**；**证据是模拟的**（未真连外部 API）；**不是 Nous Hermes 框架本身**，是其"调查 Agent 模式"。

## 十、自我进化（Agent 发现新信号）

点信号台账"运行 Agent 自学习"→ 模拟 Hermes 从已确认欺诈案例中学出一个尚未纳入规则集的新模式（如"大额付款 + 地理/行为风险"）→ 作为**候选信号**加入台账（标"Agent 发现/待验证"）→ 走每月校准验证 → 达标才转正。

```
自我进化（黑盒，会学）          统计闸门（白盒，可审计）        人工
─────────────────             ──────────────────            ────
从经验发现新模式          →    候选信号进台账（待验证）   →   PPV+置信下界校验  →  委员会批准  →  才影响分级
```

**关键原则**：自我进化只能**发现假设**，绝不直接改告警等级。任何影响分级的变更，都必须经过可审计的 PPV 校准 + 人工审批 —— 既发挥 Hermes 自我进化的价值，又守住 AML 可解释/可审计的合规底线。

## 十一、超时管理（SLA）

按 Tier 时限（T1=24h、T2=3 天）+ 付款日期 + 是否已复核，自动判定超时。
未复核且超时的告警在**付款列表、待复核队列、案例详情**中红色高亮 + `已超时` 标签；首页有"已超时"指标卡。

## 十三、创新点

1. **分层告警榨干高精度规则** —— 把 30% PPV 的硬规则用跃迁式分层（T1/T2/T3）顶上来，绝不被加权平均稀释。这是针对"规则珍贵、标签稀缺"现状的最优解。
2. **双入口统一队列** —— 付款级硬规则 + 供应商级行为信号合并成一份工作清单，抓单笔异常的同时抓拆分/突增这类跨笔模式。
3. **PPV 驱动 + Wilson 置信下界的信号校准** —— 在每年 <10 正样本的极端场景下，用置信区间下界做统计严谨的信号升降级，不被小样本噪声误导。
4. **闭环反馈实时反哺** —— 每次法务复核都结构化沉淀，实时重算信号 PPV，让"该看谁"越来越准。
5. **主从评分** —— Tier 主导决策、加权分退守 T3 灰色地带，解决了"两套分类打架"和"强信号被稀释"。
6. **AI 调查 Agent 与人工复核一体化** —— AI 多源取证 + 综合建议，人做最终判断，一次提交，证据自动进复核记录提升反馈质量。
7. **自我进化在治理闸门内** —— Agent 从经验发现新信号假设，但只能进"候选 → 校准 → 审批"的可审计闸门，不能黑盒改分级。这是 AI 自进化与合规可审计的正确结合方式。
8. **Precision@K 的目标框架** —— 放弃不可达的"识别所有洗钱"，改为可度量的"top K 含金量"，匹配业务现实。

## 十七、一句话总览

> 在"海量付款 + 极少标签 + 高精度规则"的现实下，系统放弃"识别所有洗钱"，改为用 **Tier 分层把高价值告警顶上来**（双入口：付款级硬规则 + 供应商级行为），用加权分守 T3 盲区，用**信号台账 + Wilson 置信下界校准**把每条信号的真实 PPV 测出来驱动升降级，并让 **Hermes 自我进化在可审计闸门内发现新信号**。整个系统是一个 **检测→调查→裁决→学习→回流** 的自我进化闭环 —— Legal 的每一次复核都是燃料，让"该看谁"越来越准。
